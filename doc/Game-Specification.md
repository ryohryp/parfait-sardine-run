# パフェイワ（Parfait Sardine Run）ゲーム仕様書

- バージョン: 0.2
- 更新日: 2026-07-20
- 対象: `ParfaitSardineRun`
- ステータス: コアループ再設計・実装前

---

## 1. 目的

本作を、単に障害物を避けながらスコアアイテムを集める60秒ランゲームから、**注文された材料を見極め、必要な材料を集めてパフェを完成させる判断型ランゲーム**へ発展させる。

継続プレイの主な理由を、ランキング、ガチャ、実績などの外部報酬だけに依存させない。1プレイ中に短い判断と達成を繰り返し、ゲーム本体そのものに「もう一度遊びたい」と感じる変化を作る。

---

## 2. 現状の課題

現行ゲームは、次の行動を60秒間繰り返す構成である。

- 障害物をジャンプ、スライド、ダッシュで回避または破壊する
- サーディン、チェリー、スターを取得する
- コンボとフィーバーを維持してスコアを伸ばす

操作感と演出は成立しているが、プレイヤーの最適行動が「取れるものは取り、障害物を避ける」に収束しやすい。数回のプレイで判断の変化が少なくなり、ランキングや収集要素を追加してもゲーム本体の単調さを解消できない。

---

## 3. コアコンセプト

> 注文を確認し、走りながら必要な材料を選んで集め、制限時間内にできるだけ多くのパフェを完成させる。

ゲーム中に数秒単位で次の判断を発生させる。

- 目の前の材料は現在の注文に必要か
- 材料を狙うために危険な位置へ移動するか
- 障害物を安全に避けるか、ダッシュで突破して材料を取りに行くか
- 現在の注文完成を優先するか、コンボやフィーバーを優先するか

操作を増やすのではなく、既存操作に対する判断の意味を増やす。

---

## 4. 基本ゲームループ

1. ゲーム開始時にパフェ注文を1件提示する
2. 注文には必要な材料を3個表示する
3. プレイヤーは走りながら対象材料を集める
4. 必要な材料を取得すると注文進捗が更新される
5. 3個すべてそろうとパフェが完成する
6. 完成ボーナスと注文コンボを加算する
7. 次の注文を即座に提示する
8. 60秒終了またはライフ消失まで繰り返す
9. 完成数、スコア、自己ベスト、ランキングをリザルトへ反映する

---

## 5. 1プレイの流れ

### 5.1 開始前

- メニューからゲームを開始する
- 既存のカウントダウンを維持する
- 最初の注文をカウントダウン中に表示する

### 5.2 プレイ中

- 制限時間は現行どおり60秒とする
- 画面上部に現在の注文と材料の取得状況を常時表示する
- 障害物、サーディン、フィーバー、コンボなど既存要素を維持する
- 材料取得と注文完成をプレイの主目的にする

### 5.3 終了

次のいずれかでランを終了する。

- 60秒経過: `clear`
- ライフが0になる: `crash`

ラン終了時は既存のライフ、スコア、サーディン数、距離、ベストスコアに加え、将来的に注文完成数と最大注文コンボを記録できる構造とする。

初回実装では既存ランキング送信境界を変更せず、注文システムの手応えを先に検証する。

---

## 6. プレイヤー操作

既存操作を維持する。

| 操作 | キーボード | 役割 |
| --- | --- | --- |
| ジャンプ | Space / ↑ / W | 地上障害物の回避、高所材料の取得 |
| スライド | ↓ / S | 空中障害物の回避、着地の早期化 |
| ダッシュ | → / D / Shift | 障害物の突破、材料取得への位置調整 |
| ポーズ | Escape / P | 一時停止と再開 |

初回実装では新しい操作ボタンを追加しない。

---

## 7. 注文システム

### 7.1 注文構成

初回実装では、1注文につき異なる材料3種類を要求する。

例:

- イチゴ + プリン + クリーム
- バナナ + イチゴ + クリーム
- プリン + バナナ + イチゴ

同一材料の複数要求は初回実装の対象外とする。

### 7.2 注文生成

- 材料候補から重複なしで3種類をランダム選択する
- 直前と完全に同じ注文は連続させない
- ゲーム開始時と注文完成直後に生成する
- 注文生成はプレイ中のフレームレートや描画処理から分離し、テスト可能な関数にする

### 7.3 注文進捗

- 未取得材料は未完了状態で表示する
- 対象材料を取得すると完了状態へ切り替える
- 取得済み材料を再取得しても同じ注文の進捗は増えない
- 対象外材料は注文進捗と注文コンボに影響させない
- 初回実装では対象外材料によるライフ減少や減点を行わない

### 7.4 注文完了

必要材料がすべてそろった時点で注文を完成扱いにする。

- 完成演出を短時間表示する
- 完成ボーナスを加算する
- 注文完成数を1増やす
- 注文コンボを1増やす
- 次の注文を即時生成する
- プレイ進行を長時間停止しない

完成演出は0.5〜1.0秒程度とし、操作不能な長い演出は避ける。

---

## 8. 材料システム

### 8.1 初期材料

初回実装の材料は次の4種類とする。

| ID | 表示名 | 視認性の方針 |
| --- | --- | --- |
| `strawberry` | イチゴ | 赤色、果実形状 |
| `pudding` | プリン | 黄色、カップ形状 |
| `cream` | クリーム | 白色、ホイップ形状 |
| `banana` | バナナ | 黄色、湾曲形状 |

色だけに依存せず、形状とアイコンでも識別できるようにする。

### 8.2 出現

- 既存のピックアップ出現処理を拡張する
- プレイヤーが現在の注文を完成できない状態を長時間作らない
- 注文に必要な未取得材料は、対象外材料より出現確率を高くする
- 同じ材料だけが連続しすぎないよう簡単な偏り防止を行う
- 高所と地上の両方に出現させ、既存のジャンプとスライドを活用する

### 8.3 取得判定

- プレイヤーと材料の衝突時に取得する
- 現在の注文に必要な未取得材料なら注文進捗を更新する
- 取得済みまたは対象外の材料は注文ボーナスを発生させない
- 初回実装では材料スロットを設けない

---

## 9. スコアとコンボ

### 9.1 スコア構成

総合スコアは、既存スコアに注文関連スコアを加えたものとする。

- 走行時間と距離
- 障害物回避、接近回避、ダッシュ破壊
- サーディン、チェリー、スター
- パフェ注文完成ボーナス
- 注文コンボボーナス

### 9.2 注文完成ボーナス

初期値は実装時に調整可能な定数として管理する。

推奨初期式:

```text
注文完成スコア = 基本点 1,000 × 注文コンボ倍率
```

### 9.3 注文コンボ

- 注文を完成するたびに1段階増加する
- 初回実装では対象外材料の取得で途切れない
- 障害物へ衝突した場合は1へ戻す
- 上限は調整可能な定数とする
- 既存の回避・取得コンボとは別の値として扱う

注文コンボと既存コンボを最初から統合すると調整が難しくなるため、内部状態は分離する。

---

## 10. 既存要素との統合

### 10.1 障害物とライフ

- クレート、フォーク、鳥を維持する
- ライフ3を維持する
- 衝突時の無敵時間、画面揺れ、パーティクルを維持する
- 注文材料の配置と障害物配置が理不尽に重ならないよう調整する

### 10.2 サーディン

サーディンは材料ではなく、既存のサブ収集物として維持する。

- スコアとフィーバーゲージを増やす
- 注文進捗には使用しない
- 後続フェーズでは、材料吸引や誤取得救済などの能力資源に発展可能とする

### 10.3 サーディンフィーバー

- 既存のフィーバーゲージと7秒間のフィーバーを維持する
- フィーバー中の無敵、速度上昇、ピックアップ吸引を維持する
- 材料も吸引対象に含める
- 注文に必要な材料を取りやすくする爽快区間として位置付ける

### 10.4 チェリーとスター

初回実装では既存のボーナスアイテムとして維持する。

材料との見分けが難しい場合は、出現頻度または表示サイズを調整する。

### 11.1 常時表示

- 残り時間
- スコア
- ライフ
- 既存コンボ
- サーディン数
- フィーバーゲージ
- 現在の注文
- 注文進捗
- 注文コンボ
- 完成したパフェ数

### 11.2 注文表示例

```text
ORDER  🍓 イチゴ  🍮 プリン  ☁ クリーム
        ✓          ・          ✓

PARFAIT COMBO ×3    COMPLETE 4
```

### 11.3 フィードバック

- 正しい材料取得時に対象アイコンを点灯する
- 注文完成時に `PERFECT PARFAIT!` を表示する
- 完成ボーナスと注文コンボ増加を表示する
- 次の注文への切り替えは一目で分かるが、画面全体を長く覆わない

## 12. リザルトとランキング

### 12.1 リザルト表示

初回実装では次を表示する。

- 最終スコア
- ベストスコア
- パフェ完成数
- 最大注文コンボ
- サーディン取得数
- 走行距離
- `もう一度` ボタン
- ランキングを開くボタン

### 12.2 ランキング

- 既存の読み取り専用ランキングモーダルを維持する
- プレイ中はランキングを開けない
- ゲームコンポーネントは常時マウントし、モーダル操作で再初期化しない
- 初回実装ではランキング基準を総合スコアのまま維持する
- 注文完成数ランキングは後続検討とする

### 12.3 ランライフサイクル

既存イベント境界を維持する。

- `psr:run-started`
- `psr:run-completed`

初回実装では既存の `RunSnapshot` 互換性を壊さない。注文統計を外部送信する場合は、スキーマバージョン更新または後方互換な拡張を別タスクで行う。

## 14. 段階的な拡張

### Phase 1: 注文コアループ

初回実装範囲を完成させ、次を確認する。

- 材料を見て取るか見送るか考える瞬間が生まれたか
- 注文完成が60秒中に複数回発生するか
- 注文HUDを見ながらでも障害物へ対応できるか
- 直後にもう一度遊びたくなるか

### Phase 2: リスクと選択

- 材料スロット
- 対象外材料の処理
- 危険ルートと安全ルート
- 注文の難易度差
- 簡単な注文と高得点注文の選択

### Phase 3: 完成演出と能力

- 2〜3秒以内の盛り付けアクション
- サーディンスキル
- レア材料
- 特殊注文
- ランダムイベント

### Phase 4: 継続プレイ

コアループの面白さを確認した後に追加する。

- 称号
- 外見アイテム
- 図鑑
- デイリーチャレンジ
- シーズンランキング
- 自己ベストゴースト

## 15. バランス目標

初期調整の目安とし、プレイテスト結果で変更する。

| 項目 | 目標 |
| --- | --- |
| 1プレイ | 60秒 |
| 初回注文完成 | 開始後8〜15秒 |
| 平均完成数 | 3〜6個 |
| 上手いプレイヤーの完成数 | 7個以上 |
| 注文切替演出 | 1秒以内 |
| 材料の識別 | 出現後0.5秒以内に判別可能 |
| 操作不能演出 | 原則なし |

注文を完成できない時間が長いと単なる運要素になる。必要な未取得材料の出現補正を入れ、技術と判断が結果へ反映される状態を優先する。

---

## 16. 受け入れ条件

初回実装は次をすべて満たした時点で完了とする。

1. ゲーム開始時に重複しない材料3種類の注文が表示される
2. 対象材料を取得すると対応する進捗だけが完了する
3. 同じ材料を再取得しても進捗が二重加算されない
4. 3種類すべて取得すると注文完成が一度だけ発生する
5. 注文完成スコアと注文コンボが加算される
6. 完成後に前回と完全一致しない次の注文が表示される
7. 注文処理を追加してもジャンプ、スライド、ダッシュ、障害物、ライフ、フィーバーが動作する
8. ポーズ中は注文状態とプレイ時間が進行しない
9. ランキングモーダルの開閉でゲームが再初期化されない
10. 既存のラン開始・完了イベントが重複発火しない
11. 注文生成と進捗判定に自動テストがある
12. 60秒のプレイを複数回行い、進行不能な注文が発生しない

## 17. 実装設計方針

注文ロジックをCanvas描画やReact表示へ直接埋め込まず、純粋関数として分離する。

想定する最小モデル:

```ts
type IngredientKind = 'strawberry' | 'pudding' | 'cream' | 'banana';

type ParfaitOrder = Readonly<{
  id: string;
  required: readonly IngredientKind[];
  collected: readonly IngredientKind[];
}>;

type OrderProgress = Readonly<{
  completed: boolean;
  newlyCollected: IngredientKind | null;
  order: ParfaitOrder;
}>;
```

最低限、次の責務を分離する。

- 注文生成
- 材料取得による進捗更新
- 注文完成判定
- 注文スコア計算
- Canvas内の材料出現と衝突
- React HUD表示

小さく実装し、各段階でプレイ可能な状態を維持する。

---

## 18. 成功判定

この改訂の成功は、機能数ではなくプレイ中の判断が増えたかで判断する。

最低限、プレイテストで次が確認できることを目標とする。

- 注文を見るために視線を動かす
- 目の前の材料を取るか見送るか判断する
- 必要材料を取るためにジャンプやダッシュの使い方を変える
- パフェ完成直後に次の注文へ意識が切り替わる
- リザルト後に、次はもっと多く完成できそうだと感じる

この手応えが得られない場合、ガチャや実績を追加する前に、材料配置、注文難易度、出現補正、操作感を見直す。
